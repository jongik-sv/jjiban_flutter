# /qa:integration - 통합 테스트케이스 작성 (v1)

> **최적화된 통합 테스트**: Task의 전체 워크플로우를 검증하는 통합 테스트케이스를 효율적으로 생성합니다.

## 🎯 최적화 목표
- **토큰 효율성**: 35% 토큰 절약
- **실용적 테스트**: 실제 업무 시나리오 기반 테스트
- **End-to-End**: 전체 워크플로우 검증
- **자동화 준비**: 향후 자동화 가능한 테스트 설계

## 트리거
- Task 개발이 완전히 완료된 경우
- 시스템 간 연동 테스트가 필요한 경우
- 실제 업무 시나리오 검증이 필요한 경우

## 사용법
```bash
/qa:integration "Task-3-1"
/qa:integration "Task 3.2"
/qa:integration "Task 1.개발환경구성-및-기반설정"
```

## ⚠️ 사전 검증 (Pre-execution Validation)

**실행 전 필수 체크**:
이 명령어를 실행하기 전에 다음 파일들의 존재 여부를 **반드시** 확인하세요:

### 공통 함수 (필수)
- [ ] `@docs/common/07.functions/analyze_project_context.md`

### 프로젝트 설정 (필수)
- [ ] `@docs/common/01.config/{project}/testing.md`

**검증 절차**:
1. 각 파일에 대해 Read 도구를 사용하여 존재 확인
2. **파일이 존재하지 않으면 아래 에러 메시지 출력 후 즉시 중단**
3. 모든 파일이 존재하는 경우에만 명령어 실행 진행

**검증 실패 시 에러 메시지**:
```
❌ 명령어 실행 중단

필수 파일이 존재하지 않습니다:
- {missing_file_path}

조치 방법:
1. 파일이 올바른 위치에 있는지 확인하세요
2. 누락된 파일을 생성하거나 복원하세요
3. 파일 경로가 정확한지 확인하세요

명령어 실행을 중단합니다.
```

## 자동 실행 플로우

### 1단계: 통합 테스트 범위 분석
**Auto-Persona**: quality-engineer
**MCP**: sequential

**자동 실행 단계**:
1. **공통 문서 분석 함수 호출**: `analyze_project_context(task_name)`
   - 📋 함수 참조: `@docs/common/07.functions/analyze_project_context.md`
   - 프로젝트 정보 및 Task 상세 정보 수집
   - 통합 테스트 관련 문서 자동 수집
   - 통합 테스트 범위 및 전략 수립
2. **Task 관련 문서 분석**:
   - 상세설계서에서 통합 포인트 식별
   - 구현보고서에서 실제 구현 범위 확인
   - UI 테스트케이스에서 단위 기능 확인
3. **통합 테스트 범위 정의**:
   - Frontend-Backend 연동 테스트
   - 데이터베이스 연동 테스트
   - 외부 시스템 연동 테스트 (해당 시)
   - 사용자 워크플로우 테스트

### 2단계: 비즈니스 시나리오 기반 테스트케이스 작성
**Auto-Persona**: quality-engineer
**MCP**: sequential

**자동 실행 단계**:
1. **주요 사용자 시나리오 식별**:
   - 프로젝트 PRD에서 핵심 사용자 여정 추출
   - 업무 프로세스 기반 시나리오 정의
   - 예외 상황 및 에러 케이스 식별
2. **End-to-End 테스트케이스 설계**:
   - 사용자 관점의 완전한 워크플로우
   - 각 단계별 입력 데이터 및 예상 결과
   - 성능 및 사용성 검증 포인트
3. **시스템 연동 테스트케이스 작성**:
   - API 연동 테스트 (Request/Response 검증)
   - 데이터 무결성 테스트 (CRUD 전 과정)
   - 동시성 및 부하 테스트 시나리오

### 3단계: 자동화 테스트 설계 (E2E 테스트 기반)
**Auto-Persona**: quality-engineer
**MCP**: sequential

**자동 실행 단계**:
1. **E2E 테스트 시나리오 작성**:
   - 📋 참조: `@docs/common/01.config/{project}/testing.md`
   - 사용자 인터랙션 자동화 스크립트
   - 화면 간 네비게이션 테스트
   - 폼 입력 및 검증 자동화
2. **데이터 기반 테스트 준비**:
   - 테스트 데이터 세트 생성
   - 데이터베이스 상태 설정/복원 스크립트
   - 테스트 환경 격리 방안
3. **검증 및 리포팅 설정**:
   - 자동화된 결과 검증 로직
   - 스크린샷 및 로그 수집 설정
   - 테스트 결과 리포트 생성

### 4단계: 테스트케이스 실행 및 결과 분석
**Auto-Persona**: quality-engineer
**MCP**: sequential

**자동 실행 단계**:
1. **매뉴얼 테스트 실행**:
   - 비즈니스 시나리오 기반 수동 테스트
   - 사용성 및 접근성 검증
   - 크로스 브라우저 테스트
2. **자동화 테스트 실행**:
   - Playwright 스크립트 실행
   - 성능 및 부하 테스트 수행
   - 회귀 테스트 실행
3. **결과 분석 및 리포팅**:
   - 테스트 결과 종합 분석
   - 발견된 이슈 분류 및 우선순위 설정
   - 테스트 커버리지 분석

### 5단계: 통합 테스트 보고서 작성
**Auto-Persona**: technical-writer
**MCP**: sequential

**자동 실행 단계**:
1. **테스트 결과 종합**:
   - 모든 테스트 케이스 실행 결과 정리
   - 통과/실패 통계 및 트렌드 분석
   - 품질 메트릭 계산 (결함 밀도, 커버리지 등)
2. **이슈 및 개선사항 정리**:
   - 발견된 결함 목록 및 해결 상태
   - 성능 이슈 및 최적화 권장사항
   - 사용성 개선 제안사항
3. **통합 테스트 보고서 작성**:
   - 테스트 범위 및 방법론 기술
   - 상세한 테스트 결과 및 분석
   - 품질 평가 및 릴리즈 준비도 판정

## 테스트 품질 기준

### ✅ 커버리지 기준
- **기능 커버리지**: 요구사항 90% 이상
- **시나리오 커버리지**: 주요 사용자 여정 100%
- **API 커버리지**: 모든 엔드포인트 검증
- **UI 커버리지**: 모든 화면 및 상호작용 테스트

### 📈 성능 기준
- **응답 시간**: API 응답 < 2초
- **페이지 로딩**: 초기 로딩 < 3초
- **동시 사용자**: 50명 동시 접속 지원
- **메모리 사용량**: 브라우저 메모리 < 100MB

### 🎯 품질 게이트
- **치명적 결함**: 0개
- **높은 우선순위 결함**: ≤ 2개
- **테스트 통과율**: ≥ 95%
- **자동화율**: ≥ 70%
